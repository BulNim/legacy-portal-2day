# 4-4. 코드베이스 구조 파악

`legacy-portal` 코드베이스의 패키지 구조와 각 서비스의 책임을 정리한다.
리팩토링을 시작하기 전, "지금 무엇이 어디에 있고 무슨 일을 하는가"를 먼저 고정해두는 문서.

## 1. 전체 구조

```
com.ktds.portal
├── PortalApplication.java     — 부트 진입점
├── DataSourceConfig.java      — MariaDB 우선, 접속 실패 시 H2로 자동 전환
├── DataSeeder.java            — 데모용 사용자 4명 초기 적재
├── approval/                  — 결재 도메인
│   ├── Approval.java              (Entity)
│   ├── ApprovalRepository.java    (Repository)
│   ├── ApprovalService.java       (Service)
│   └── ApprovalController.java    (Controller)
├── notice/                    — 공지 도메인
│   ├── Notice.java
│   ├── NoticeRepository.java
│   └── NoticeService.java         ※ Controller 없음
├── schedule/                  — 일정 도메인
│   ├── Schedule.java
│   ├── ScheduleRepository.java
│   └── ScheduleService.java       ※ Controller 없음
├── user/                      — 사용자 도메인
│   ├── User.java
│   ├── UserRepository.java
│   └── UserController.java        ※ Service 없음, Controller가 Repository 직접 호출
└── common/                    — 도메인 횡단 인프라
    ├── SmtpMailSender.java        메일 발송 (콘솔 출력으로 대체)
    └── FileAuditLogger.java       감사 로그 (콘솔 출력으로 대체)
```

각 도메인 폴더는 `Entity + Repository + Service (+ Controller)` 패턴을 따르되,
**도메인마다 갖춘 레이어가 다르다** (아래 3절 참고). 테스트 코드(`src/test/java`)는
프로젝트 전체에 **단 하나도 없다** — 안전망이 없는 상태에서 리팩토링을 시작해야 하는 실습 조건.

## 2. 도메인별 서비스 책임

### 2.1 approval (결재) — 이 코드베이스의 "주인공"

- **엔티티**: `Approval` — 빈약한 도메인 모델. `title/content/type/status/priority/drafterId/approverId/rejectReason/amount` 를 모두 public getter/setter로 노출.
  - `status`(int): 0=임시저장, 1=상신, 2=승인, 3=반려, 9=취소
  - `type`(int): 1=지출, 2=휴가, 3=구매, 4=기타
  - `priority`(int): 1=낮음, 2=보통, 3=높음
- **서비스(`ApprovalService`)가 떠안은 책임**:
  - 결재 문서 생성(`create`) — 파라미터 8개, 상태 초기화, 감사 로그 기록까지 한 메서드에서 처리
  - 상태 전이 처리(`processApproval`) — **상신/승인/반려/취소 4가지 액션을 하나의 메서드가 action 코드(1/2/3/9)로 분기**. 각 분기 안에서 상태 가드 검증 + 권한 판정(`role >= 2`) + 금액 기준 우선순위 자동 상향 + 저장 + 메일 발송 + 감사 로그를 모두 수행
  - 화면 표시용 라벨 변환(`statusLabel`), 금액 등급 산정(`amountGrade`) — 화면/도메인 로직까지 서비스가 겸함
  - 협력 객체(`SmtpMailSender`, `FileAuditLogger`)를 생성자 주입이 아닌 **직접 `new`** 로 생성 (DI 없음 → 테스트 시 대체 불가)
- **컨트롤러(`ApprovalController`)**: `Map<String,Object>` 로 요청을 받아 그대로 서비스에 전달. 요청 DTO·검증 없음. `action`(1/2/3/9) 매직넘버를 API 계약으로 그대로 노출.
- **권한 판정 규칙(레거시 동작, 보존 대상)**: 결재자 본인이 아니면 승인/반려 조용히 무시. `role < 2`(사원)이면 승인/반려 권한 없이 조용히 무시(예외 없음).

### 2.2 notice (공지)

- **엔티티**: `Notice` — `status`(0=임시, 1=게시, 9=내림), `category`(1=일반, 2=긴급, 3=인사), `pinned`(boolean).
- **서비스(`NoticeService`)**:
  - 공지 생성(`create`) — `ApprovalService.create()` 와 사실상 동일한 구조의 감사 로그 코드가 복붙되어 있음
  - 게시 처리(`publish`) — `role >= 2` 권한 판정(`ApprovalService`의 승인 권한 판정과 동일한 조건 복붙) + 상태 가드(`status==0`일 때만) + **긴급(category==2) 공지면 전 직원에게 메일 발송**(`userRepo.findAll()` 순회) + 감사 로그
  - `statusLabel` — `ApprovalService.statusLabel` 과 구조가 거의 동일(중복)
- **컨트롤러 없음** — HTTP로 노출되지 않은 도메인. `NoticeService`는 현재 API 표면에 연결되어 있지 않다.
- `SmtpMailSender`, `FileAuditLogger` 를 여기서도 직접 `new` (강결합 반복).

### 2.3 schedule (일정)

- **엔티티**: `Schedule` — `status`(0=예정, 1=확정, 9=취소), `startAt`/`endAt`/`location`.
- **서비스(`ScheduleService`)**:
  - 일정 생성(`create`) — 제목/시작·종료시각 검증 + **같은 소유자(`ownerId`)의 기존 일정과 시간 겹침을 직접 루프 돌며 확인**(`flag1` 이라는 이름 없는 boolean으로 결과 누적) + 저장 + 감사 로그. 겹치면 원인 설명 없이 `null` 반환.
  - 확정 처리(`confirm`) — 본인 소유 + `status==0`(예정)일 때만 `status=1`(확정)로 전이 + 감사 로그
  - 메일 발송 없음(`SmtpMailSender` 미사용) — `notice`/`approval` 과 달리 감사 로그만 기록
- **컨트롤러 없음** — `notice` 와 마찬가지로 API 미노출 상태.

### 2.4 user (사용자)

- **엔티티**: `User` — `role`(int, 1=사원/2=팀장/3=임원)이 다른 세 도메인의 권한 판정에 매직넘버로 그대로 쓰인다.
- **서비스 없음** — `UserController` 가 `UserRepository` 를 직접 호출(`findAll()`)해 전체 사용자 목록만 반환. 데모 프론트엔드의 "로그인(사용자 선택)" 화면 용도.
- `DataSeeder` 가 앱 최초 기동 시 사용자 4명(사원 2명·팀장 1명·임원 1명)을 시드 데이터로 적재.

### 2.5 common (도메인 횡단 인프라)

- `SmtpMailSender` — 실제 SMTP 대신 콘솔 출력으로 대체된 메일 발송기. 인터페이스 없이 구체 클래스로만 존재하며 `approval`/`notice` 서비스가 각각 직접 `new`.
- `FileAuditLogger` — 실제 파일 append 대신 콘솔 출력으로 대체된 감사 로그 기록기. `approval`/`notice`/`schedule` 세 서비스 모두 직접 `new`, 로그 포맷(`[yyyy-MM-dd HH:mm:ss] ACTION id=.. by=..`)이 서비스마다 복붙되어 있다.

## 3. 도메인별 레이어 비대칭 (현재 상태)

| 도메인 | Entity | Repository | Service | Controller | 비고 |
|---|---|---|---|---|---|
| approval | ✅ | ✅ | ✅ | ✅ | API로 완전히 노출, 4개 레이어 모두 존재 |
| notice | ✅ | ✅ | ✅ | ❌ | 서비스 로직은 있으나 API 미노출 |
| schedule | ✅ | ✅ | ✅ | ❌ | 서비스 로직은 있으나 API 미노출 |
| user | ✅ | ✅ | ❌ | ✅ | Service 계층 없이 Controller가 Repository 직접 호출 |

## 4. API 표면 (현재 노출된 엔드포인트)

- `POST /api/approvals` — 결재 생성
- `POST /api/approvals/{id}/process` — 결재 처리(상신/승인/반려/취소, `action` 코드로 분기)
- `GET /api/approvals/drafts/{userId}` — 내가 기안한 결재 목록
- `GET /api/approvals/inbox/{userId}` — 내가 결재할 목록
- `GET /api/users` — 전체 사용자 목록

`notice`, `schedule` 도메인은 서비스 로직은 존재하지만 컨트롤러가 없어 API로는 아직 접근할 수 없다.

## 5. 도메인 간 의존 관계

- `approval`, `notice`, `schedule` 세 서비스 모두 `user.UserRepository` 에 의존(권한 판정·메일 수신자 조회 목적) — `user` 가 사실상 공통 참조 대상.
- `approval`, `notice` 서비스는 `common.SmtpMailSender` + `common.FileAuditLogger` 에 의존.
- `schedule` 서비스는 `common.FileAuditLogger` 만 의존(메일 미사용).
- 도메인 간 서비스-서비스 직접 호출은 없음(예: `NoticeService` 가 `ApprovalService` 를 참조하지 않음) — 결합은 전부 `common`/`user` 를 통한 간접 결합.

## 6. 반복되는 구조적 패턴 (리팩토링 대상)

세 도메인 서비스(`ApprovalService`/`NoticeService`/`ScheduleService`)에 걸쳐 동일한 패턴이 반복된다.

1. **감사 로그 복붙**: `LocalDateTime.now().format(...)` + `audit.write(...)` 형태의 6줄 블록이 각 서비스의 모든 상태 변경 메서드마다 거의 동일하게 반복.
2. **직접 `new` 강결합**: `SmtpMailSender`/`FileAuditLogger` 를 생성자 주입 없이 필드에서 직접 생성 → 테스트 시 대체 불가.
3. **`role >= 2` 권한 판정 복붙**: `ApprovalService`(승인/반려)와 `NoticeService`(게시)에서 동일한 조건이 반복.
4. **`statusLabel` 류 매핑 메서드 중복**: 상태 코드를 한글 라벨로 바꾸는 if-else 사슬이 `Approval`/`Notice` 양쪽에 거의 동일한 구조로 존재.
5. **원시 타입 집착**: `status`/`type`/`priority`/`category`/`role` 이 전부 int이며, 의미는 주석에만 존재하고 DB에는 정수 그대로 저장된다(CLAUDE.md 불변 규칙에 따라 리팩토링 후에도 DB 저장값은 정수로 유지해야 함).
6. **조용한 실패**: 권한 없음/상태 불일치/시간 겹침 등 실패 상황에서 예외 대신 `return`(void) 또는 `null` 을 반환 — 호출자가 실패 원인을 알 수 없는 레거시 동작이 세 서비스에 공통.

이 문서는 구조 스냅샷이며, 실제 리팩토링 방향·불변 규칙은 프로젝트 루트의 `CLAUDE.md` 를 따른다.
