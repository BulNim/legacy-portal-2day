# 4-13. processApproval() 상태 전이 규칙

`ApprovalService.processApproval(id, userId, action, reason)`(75-160행)의 4개 action 분기를
전제조건·결과 상태·부수효과 기준으로 표로 정리한다. [4-7](./4-7.%20긴%20메서드%20탐지.md) 에서
분석한 중첩 구조, [4-8](./4-8.%20매직넘버%20탐지.md) 의 코드값 의미를 상태 전이 관점에서 재구성한 것이며,
`docs/legacy-portal_사용자테스트_보고서.docx` 의 Playwright 시나리오 1·2·4·5가 이 표의 상신/승인/반려/
권한없음 행을 실제 화면에서 검증한 결과다.

## 상태 전이표

| action | 동작 | 전제조건 (모두 만족해야 전이) | 결과 상태(status) | 부수효과 | 조건 불만족 시 |
|:---:|---|---|:---:|---|---|
| **1** | 상신(SUBMIT) | ① 대상 결재 존재<br>② `status == 0`(임시저장) | `1` (상신) | · `type==1`(지출) `&&` `amount>=1,000,000` 이면 `priority=3`(높음)로 자동 상향<br>· 결재자에게 메일 발송<br>· 감사 로그 `APPROVAL SUBMIT` 기록 | 조용히 무시(예외 없음, 아무 것도 바뀌지 않음) |
| **2** | 승인(APPROVE) | ① 대상 결재 존재<br>② 처리자(`userId`) 존재<br>③ `status == 1`(상신)<br>④ `approverId == userId`(결재자 본인)<br>⑤ 처리자 `role >= 2`(팀장 이상) | `2` (승인) | · 기안자에게 메일 발송<br>· 감사 로그 `APPROVAL APPROVE` 기록 | 조용히 무시 — ③~⑤ 중 하나라도 불만족 시 예외·에러 메시지 없이 상태 유지(S18, 사용자 테스트 시나리오 5로 재현 확인) |
| **3** | 반려(REJECT) | ① 대상 결재 존재<br>② 처리자(`userId`) 존재<br>③ `status == 1`(상신)<br>④ `approverId == userId`(결재자 본인)<br>⑤ 처리자 `role >= 2`(팀장 이상) | `3` (반려) | · `rejectReason`에 `reason` 저장(미입력 시 빈 문자열 허용)<br>· 기안자에게 메일 발송(반려 사유 포함)<br>· 감사 로그 `APPROVAL REJECT` 기록 | 조용히 무시 — 승인과 동일한 조건(③~⑤), 동일하게 무시 |
| **9** | 취소(CANCEL) | ① 대상 결재 존재<br>② 처리자(`userId`) 존재<br>③ `status == 0`(임시저장) `\|\|` `status == 1`(상신)<br>④ `drafterId == userId`(기안자 본인) | `9` (취소) | · 메일 발송 없음<br>· 감사 로그 `APPROVAL CANCEL` 기록 | 조용히 무시 — ③·④ 불만족 시 상태 유지 |
| (그 외) | 정의되지 않은 action(0, 4~8, 10 이상) | — | 변화 없음 | 없음 | 4개 `if/else if` 분기 어디에도 해당하지 않아 아무 처리도 하지 않고 메서드 종료(조용히 무시) |

## 전제조건 공통 패턴

- **대상 조회 실패**(`repo.findById(id)` 가 없음) → 모든 action에서 공통으로 즉시 `return`(76-80행).
- **처리자 조회 실패**(`userRepo.findById(userId)` 가 없음) → 모든 action에서 공통으로 즉시 `return`(81-84행).
- **승인/반려(action 2·3)**: 상태 가드(`status==1`) + 결재자 본인 확인 + 권한(`role>=2`) 3중 조건이
  완전히 동일 — [4-5. 중복 코드 탐지](./4-5.%20중복%20코드%20탐지.md) 4절에서 지적한 `role>=2` 중복이
  바로 이 지점.
- **상신/취소(action 1·9)**: 결재자 확인·권한 확인 없이 상태 가드 + 본인 확인(취소만)만 검사 — 상신은
  기안자 본인 여부조차 확인하지 않는다(누구든 `drafterId`를 몰라도 id만 알면 상신 가능, 레거시 결함이나
  리팩토링 단계에서는 보존 대상).

## 상태값 요약 (참고)

| status 값 | 의미 | 도달 가능한 action |
|:---:|---|---|
| 0 | 임시저장(DRAFT) | `create()` 최초 생성 시에만 부여 |
| 1 | 상신(SUBMITTED) | action=1 (전제: status 0) |
| 2 | 승인(APPROVED) | action=2 (전제: status 1) |
| 3 | 반려(REJECTED) | action=3 (전제: status 1) |
| 9 | 취소(CANCELED) | action=9 (전제: status 0 또는 1) |

2(승인)·3(반려)·9(취소)는 모두 종료 상태로, 이후 어떤 action을 호출해도 위 표의 상태 가드 조건을
만족하지 못해 항상 조용히 무시된다(전이 종점).
