# 4-5. 중복 코드 탐지

`ApprovalService` / `NoticeService` / `ScheduleService` 세 서비스에 걸쳐 반복되는 코드를
유형별로 정리한다. [4-4. 코드베이스 구조 파악](./4-4.%20코드베이스%20구조%20파악.md) 의 6절에서
지적한 "반복되는 구조적 패턴"을 실제 코드 스니펫·줄 번호 단위로 구체화한 문서.
공통 모듈 추출(Extract Method / Class) 실습의 대상 목록으로 사용한다.

## 1. 감사 로그 기록 패턴 (6곳 중복)

가장 많이 반복되는 중복. `LocalDateTime.now()` 를 포맷팅해 `audit.write(...)` 로 넘기는
동일한 2줄 블록이 서비스 3개, 총 6개 지점에 복붙되어 있다.

```java
String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
audit.write("[" + now + "] <ACTION> id=" + <id> + " by=" + <userId>);
```

| 파일 | 위치 | ACTION 문자열 |
|---|---|---|
| `ApprovalService.java` | `create()` 62-65행 | `APPROVAL CREATE` |
| `ApprovalService.java` | `writeAudit()` 163-166행 (private 헬퍼, `processApproval()`의 submit/approve/reject/cancel 4곳에서 호출) | `APPROVAL SUBMIT/APPROVE/REJECT/CANCEL` |
| `NoticeService.java` | `create()` 44-45행 | `NOTICE CREATE` |
| `NoticeService.java` | `publish()` 69-70행 | `NOTICE PUBLISH` |
| `ScheduleService.java` | `create()` 62-63행 | `SCHEDULE CREATE` |
| `ScheduleService.java` | `confirm()` 76-77행 | `SCHEDULE CONFIRM` |

**참고**: `ApprovalService` 는 `writeAudit()` 사설 헬퍼로 한 번 추출했지만 `create()` 안에는
같은 블록이 또 인라인으로 남아 있다 — "불완전한 중복 제거"의 예시(코드 내 기존 주석에서도
`[스멜4] 그나마 추출했지만 create() 안에는 또 복붙이 남아 있다`로 명시). `NoticeService`,
`ScheduleService` 는 이런 헬퍼조차 없이 매번 인라인 작성.

## 2. 협력 객체 직접 `new` (강결합, 3곳)

`SmtpMailSender`/`FileAuditLogger` 를 생성자 주입 없이 필드 초기화에서 직접 생성.
DI 컨테이너를 거치지 않으므로 테스트에서 가짜 객체로 대체할 수 없다.

| 파일 | 위치 |
|---|---|
| `ApprovalService.java` | 37-38행 `mail = new SmtpMailSender()`, `audit = new FileAuditLogger()` |
| `NoticeService.java` | 24-25행 `mail = new SmtpMailSender()`, `audit = new FileAuditLogger()` |
| `ScheduleService.java` | 20행 `audit = new FileAuditLogger()` (메일은 미사용) |

패턴 자체는 코드 중복이라기보다 **동일한 설계 결함의 반복**(구조적 중복)에 가깝다.

## 3. 엔티티 조회 + null 체크 후 조용히 리턴 (3곳)

대상 엔티티와 사용자 엔티티를 조회한 뒤 `null` 이면 예외 없이 바로 리턴하는 동일한 가드 블록.

```java
X x = repo.findById(id).orElse(null);
if (x == null) return;              // 실패 원인 소실
User u = userRepo.findById(userId).orElse(null);
if (u == null) return;
```

| 파일 | 위치 |
|---|---|
| `ApprovalService.processApproval()` | 76-84행 (`Approval`, `User` 순차 조회) |
| `NoticeService.publish()` | 50-53행 |
| `ScheduleService.confirm()` | 68-71행 |

레거시 동작(예외 대신 조용한 무시)은 CLAUDE.md 불변 규칙에 따라 리팩토링 단계에서 **그대로 보존**해야
하는 대상이지만, 가드 코드 자체의 중복은 공통 메서드로 추출할 수 있다.

## 4. 권한 판정 `role >= 2` 복붙 (3곳)

"역할이 팀장(2) 이상이어야 처리 가능"이라는 동일한 조건이 승인/반려/게시 세 곳에 반복.

| 파일 | 위치 | 문맥 |
|---|---|---|
| `ApprovalService.processApproval()` | 114행 | 승인(action=2) 처리 조건 |
| `ApprovalService.processApproval()` | 133행 | 반려(action=3) 처리 조건 (승인과 완전히 동일한 조건식) |
| `NoticeService.publish()` | 56행 | 공지 게시 조건 |

`role`(1=사원/2=팀장/3=임원)의 의미가 세 지점 모두 주석으로만 설명되어 있고, 판정 기준(`>= 2`)이
바뀌면 세 곳을 동시에 고쳐야 한다.

## 5. 메일 본문 생성 패턴 (4곳)

"안녕하세요 {수신자}님, ... 제목: {제목}" 형태의 인사말+본문 조립 로직이 결재 3곳(상신/승인/반려
알림)과 공지 1곳(긴급 공지 전체 발송)에서 문자열 결합으로 반복된다.

```java
String body = "안녕하세요 " + <수신자>.getName() + "님,\n"
        + "<안내 문구>\n제목: " + <대상>.getTitle() ...;
mail.send(<수신자>.getEmail(), "[<구분>] " + <대상>.getTitle(), body);
```

| 파일 | 위치 | 발송 트리거 |
|---|---|---|
| `ApprovalService.processApproval()` | 101-107행 | 상신 시 결재자에게 |
| `ApprovalService.processApproval()` | 118-124행 | 승인 시 기안자에게 |
| `ApprovalService.processApproval()` | 138-144행 | 반려 시 기안자에게(반려 사유 포함) |
| `NoticeService.publish()` | 62-67행 | 긴급 공지 게시 시 전 직원에게(`userRepo.findAll()` 순회) |

본문 문구는 각기 다르지만 "인사말 조립 → 제목 삽입 → `mail.send()` 호출"이라는 **구조**가 동일하게
반복되며, 제목(`subject`)도 `"[구분] " + title` 형태로 일관된 규칙을 따른다.

## 6. `statusLabel()` 상태코드 → 한글 라벨 매핑 (2곳)

상태 정수를 화면 표시용 한글 문자열로 변환하는 if-else 사슬이 `Approval`/`Notice` 양쪽에
거의 동일한 구조로 존재한다(값 집합만 다름).

| 파일 | 위치 | 매핑 |
|---|---|---|
| `ApprovalService.statusLabel()` | 169-179행 | 0→임시저장, 1→상신, 2→승인, 3→반려, 9→취소, else→알수없음 |
| `NoticeService.statusLabel()` | 76-82행 | 0→임시, 1→게시, 9→내림, else→알수없음 |

`Notice` 의 매핑(0/1/9)이 `Approval` 매핑(0/1/2/3/9)의 부분집합과 유사한 구조를 취하고 있어,
"상태 enum이 라벨을 스스로 아는" 형태로 통합할 수 있는 후보.

`ScheduleService` 에는 `statusLabel()` 이 없다 — 세 도메인이 모두 상태 코드(int)를 가지고 있음에도
라벨 변환 로직의 존재 여부가 도메인마다 다르다는 점도 비일관성으로 기록해둔다.

## 7. 요약 — 중복 유형별 발생 횟수

| 중복 유형 | 발생 횟수 | 관련 서비스 |
|---|---|---|
| 감사 로그 기록 블록 | 6 | Approval, Notice, Schedule |
| 협력 객체 직접 `new` | 3 | Approval, Notice, Schedule |
| 조회 + null 체크 조용한 리턴 | 3 | Approval, Notice, Schedule |
| `role >= 2` 권한 판정 | 3 | Approval(x2), Notice |
| 메일 본문 조립 패턴 | 4 | Approval(x3), Notice |
| `statusLabel()` 매핑 사슬 | 2 | Approval, Notice |

`ScheduleService` 는 메일 발송과 상태 라벨 변환이 없어 다른 두 서비스보다 중복 지점 수가 적지만,
감사 로그·null 가드 패턴은 동일하게 반복된다.

## 8. 공통 모듈 추출 후보 (제안, 미구현)

리팩토링 단계에서 고려할 수 있는 추출 방향을 참고로 남긴다(CLAUDE.md의 "리팩토링 - 개선 방향"에
해당하며, 이 문서는 탐지 결과만 정리하고 실제 추출은 별도 작업에서 수행한다).

- **감사 로그**: `FileAuditLogger.write(String)` 호출부의 타임스탬프 포맷팅을 `AuditLogger` 쪽으로
  이동하거나, `audit.write(action, entityId, userId)` 형태의 시그니처로 서비스 쪽 포맷팅 중복 제거.
- **메일 본문**: 액션 타입(상신/승인/반려/긴급공지)별 템플릿을 `common` 패키지의 메일 빌더로 추출.
- **권한 판정**: `User.isManagerOrAbove()` 같은 도메인 메서드로 `role >= 2` 조건을 캡슐화(Rich Domain
  방향과 일치).
- **상태 라벨**: 각 도메인 status를 enum화하면서 라벨을 enum이 스스로 갖도록 통합(`Enum.label()`).
- **조회 + null 가드**: 제네릭 헬퍼보다는 각 서비스 메서드 내부를 Extract Method로 정리하는 편이
  가독성 측면에서 더 적합할 수 있음(공통 추출 여부는 판단 필요).
