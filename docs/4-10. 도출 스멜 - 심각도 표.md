# 4-10. 도출 스멜 - 심각도 표

[4-4](./4-4.%20코드베이스%20구조%20파악.md) ~ [4-9](./4-9.%20강결합%20탐지.md) 에서 개별적으로
탐지한 스멜을 하나의 표로 취합하고 **심각도**를 매긴다. 리팩토링 착수 순서를 정하기 위한 근거 자료.

## 심각도 기준

| 심각도 | 기준 |
|---|---|
| **상** | 여러 파일·서비스에 걸쳐 영향을 미치거나, 테스트 작성·안전한 변경 자체를 가로막는 구조적 문제 |
| **중** | 특정 서비스 내에서 반복되어 유지보수 비용을 계속 늘리는 문제(중복, 응집도 저하 등) |
| **하** | 가독성을 저해하지만 동작·확장성에는 직접적 영향이 적은 문제(네이밍, 사소한 중복) |

**요약**: 상 6건 · 중 8건 · 하 4건 (총 18건)

## 스멜 목록

| 심각도 | 위치 | 스멜 | 권장 리팩토링 |
|:---:|---|---|---|
| 상 | 프로젝트 전체 (`src/test/java` 자체가 없음) | No Tests — 테스트 코드 0건, 안전망 부재 | 특성화 테스트(Characterization Test) 작성을 리팩토링 착수 전 **최우선** 수행 |
| 상 | `ApprovalService.java` 전체(31-197행) | God Class — 검증+영속화+메일+감사로그+포맷팅+권한판정을 한 클래스가 전담 | Extract Class로 메일 알림·감사로그·라벨/등급 계산 책임 분리 |
| 상 | `ApprovalService.processApproval()` 75-160행 | Long Method(86줄) + 깊은 중첩(최대 6단계) | Extract Method로 submit/approve/reject/cancel 분리 — `processApproval(id, userId, action, reason)` public 시그니처는 유지, 내부는 위임만 ([4-7](./4-7.%20긴%20메서드%20탐지.md)) |

## 우선순위 메모

- **상 등급 6건**이 리팩토링의 실질적 진입 장벽이다. 그중에서도 **No Tests**가 선행 조건 — 나머지
  5건(God Class, Long Method, Tight Coupling, Primitive Obsession, 요청 DTO 부재)을 건드리기 전에
  현재 동작을 특성화 테스트로 고정해야 CLAUDE.md의 "리팩토링 전후 테스트 green 유지" 검증이 가능하다.
- **중 등급** 중 "조용한 실패"는 유일하게 *지금 고치면 안 되는* 항목이다 — 결함이지만 리팩토링 단계의
  불변 규칙(관찰 가능한 동작 100% 보존) 대상이므로, 이 표에서는 심각도만 기록하고 착수 대상에서는
  제외한다.
- **하 등급 4건**은 위 상/중 등급 리팩토링(Extract Method, DTO 도입, enum화)을 진행하는 과정에서
  자연스럽게 같이 해소되는 경우가 많다(예: DTO 도입 시 Long Parameter List 자동 해소).
