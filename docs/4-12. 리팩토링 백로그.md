# 4-12. 리팩토링 백로그

[docs/smells.md](./smells.md) 의 진단 결과(S01~S18)를 실행 가능한 백로그 티켓으로 변환한다.
순서는 우선순위(P1→P2→P3)와 티켓 간 의존 관계를 함께 고려해 정했다 — 단순히 P값으로만 정렬하면
`processApproval()` 을 다른 P1 정리 작업 이전에 건드리게 되어 회귀 위험이 커지므로, 4-11의
"권장 착수 순서"를 그대로 반영했다.

각 티켓은 CLAUDE.md 불변 규칙(API·시그니처·요청응답·DB 저장값·관찰 가능한 동작 불변)을 전제로 하며,
완료 조건에 "특성화 테스트 green 유지"를 공통으로 포함한다.

## 백로그

### BL-01 — 특성화 테스트 작성
- **관련 스멜**: S01 (No Tests)
- **우선순위**: P1 · **선행 작업**: 없음 (모든 후속 티켓의 전제 조건)
- **작업 내용**: `Approval`/`Notice`/`Schedule` 각 서비스의 현재 동작(정상 케이스 + 조용한 실패 케이스
  S18 포함)을 테스트로 고정한다. 상신/승인/반려/취소, 공지 게시(+긴급 메일), 일정 생성(+시간 겹침),
  권한 없는 처리가 조용히 무시되는 케이스까지 포함.
- **완료 조건(DoD)**: 리팩토링 전 코드에서 전부 green. 이후 모든 백로그 티켓은 이 테스트가 green으로
  유지되는 것을 완료 조건으로 삼는다.

### BL-02 — 매직넘버 → enum 전환
- **관련 스멜**: S03 (Primitive Obsession)
- **우선순위**: P1 · **선행 작업**: BL-01
- **작업 내용**: `Approval.status/type/priority`, `Notice.status/category`, `Schedule.status`,
  `User.role` 을 각각 enum으로 감싸고 `AttributeConverter` 로 기존 정수 DB 값을 그대로 매핑
  (`@Enumerated` 금지). 서비스 내 정수 비교문(`s==0`, `proc==2`, `role>=2` 등)을 enum 비교로 치환.
- **완료 조건**: DB에는 기존 정수(0/1/2/3/9 등)가 그대로 저장됨을 테스트로 확인. BL-01 테스트 green 유지.
- **비고**: 이후 BL-09(Long Method 분해)의 난이도를 낮추는 기반 작업이므로 P1 중에서도 먼저 진행.

### BL-03 — 협력 객체 DI 적용
- **관련 스멜**: S02 (Tight Coupling)
- **우선순위**: P1 · **선행 작업**: BL-01
- **작업 내용**: `SmtpMailSender`/`FileAuditLogger` 에 인터페이스(`MailSender`/`AuditLogger` 등) 추출,
  구현체에 `@Component` 부여, `ApprovalService`/`NoticeService`/`ScheduleService` 생성자 주입으로 전환.
- **완료 조건**: 세 서비스 모두 `new SmtpMailSender()`/`new FileAuditLogger()` 코드 제거. BL-01 테스트
  green 유지(+ 테스트에서 가짜 구현체로 대체 가능함을 확인하는 테스트 1개 이상 추가).

### BL-04 — 결재 요청 DTO 도입
- **관련 스멜**: S04 (요청 DTO 부재), S15 (Long Parameter List, 부수 해소)
- **우선순위**: P1 · **선행 작업**: BL-01
- **작업 내용**: `ApprovalController` 의 `Map<String,Object>` 파라미터를 요청 record DTO
  (`ApprovalCreateRequest`, `ApprovalProcessRequest`)로 대체 + Bean Validation 적용.
  `ApprovalService.create()` 의 8개 파라미터는 DTO 도입에 맞춰 자연히 정리(S15 동시 해소).
- **완료 조건**: 기존 JSON 요청/응답 구조·필드명·HTTP 메서드·URL 100% 동일. BL-01 테스트 green 유지.

### BL-05 — 감사 로그 중복 제거
- **관련 스멜**: S05
- **우선순위**: P1 · **선행 작업**: BL-01, BL-03(권장 — DI된 `AuditLogger` 기준으로 정리)
- **작업 내용**: `LocalDateTime.now().format(...) + audit.write(...)` 6곳을
  `audit.write(action, entityId, userId)` 형태의 단일 호출로 통합(타임스탬프 포맷팅은 로거 쪽으로 이동).
- **완료 조건**: 감사 로그 출력 포맷이 기존과 동일(문자열 비교 테스트). BL-01 테스트 green 유지.

### BL-06 — 권한 판정 캡슐화
- **관련 스멜**: S06
- **우선순위**: P1 · **선행 작업**: BL-01 (BL-02 완료 후 진행 시 `User.role` enum 기준으로 바로 작성 가능)
- **작업 내용**: `User` 에 `isManagerOrAbove()` 같은 도메인 메서드를 추가하고, `ApprovalService`
  승인/반려 분기 및 `NoticeService.publish()` 의 `role>=2` 조건 3곳을 이 메서드 호출로 치환.
- **완료 조건**: 권한 판정 결과 동일. BL-01 테스트 green 유지.

### BL-07 — `statusLabel`/`amountGrade` 도메인 이동
- **관련 스멜**: S07 (Feature Envy), S14 (statusLabel 매핑 중복, 부수 해소)
- **우선순위**: P1 · **선행 작업**: BL-01, BL-02(권장 — enum에 라벨을 붙이는 방식으로 함께 처리)
- **작업 내용**: `ApprovalService.statusLabel()`/`amountGrade()` 를 `Approval` 도메인으로 Move Method.
  BL-02에서 만든 status enum이 라벨을 스스로 갖게 하면 `NoticeService.statusLabel()` 과의 중복(S14)도
  같은 패턴으로 해소된다.
- **완료 조건**: 두 서비스의 `statusLabel()` 반환값이 기존과 동일. BL-01 테스트 green 유지.

### BL-08 — `NoticeService.publish()` 중첩 완화
- **관련 스멜**: S08
- **우선순위**: P1 · **선행 작업**: BL-01
- **작업 내용**: Guard Clause 적용(`role`/`status` 조건 역전 후 early return) + 긴급 메일 발송 부분을
  private 메서드로 Extract Method. 5단계 중첩을 2~3단계로 완화.
- **완료 조건**: `publish()` 동작(게시 조건, 긴급 메일 발송 대상) 동일. BL-01 테스트 green 유지.

### BL-09 — `processApproval()` 분해 (God Class + Long Method)
- **관련 스멜**: S09, S10
- **우선순위**: P2 · **선행 작업**: BL-01, BL-02, BL-03, BL-05, BL-06 (매직넘버·DI·감사로그·권한판정이
  먼저 정리되어 있어야 분해 작업 자체의 위험이 줄어든다)
- **작업 내용**: `processApproval(id, userId, action, reason)` **public 시그니처는 유지**한 채, 내부를
  `submit()`/`approve()`/`reject()`/`cancel()` 4개 private 메서드로 Extract Method. 금액 기준 우선순위
  자동 상향 로직은 `Approval` 도메인으로 이동(Rich Domain 방향, BL-11과 연계 가능).
- **완료 조건**: 4개 액션 각각 상태 전이 결과·메일 발송·감사 로그가 기존과 100% 동일. BL-01 테스트
  green 유지(리팩토링 전후 테스트 코드 자체는 변경 없이 그대로 통과해야 함).

### BL-10 — 메일 본문 템플릿 추출
- **관련 스멜**: S11
- **우선순위**: P2 · **선행 작업**: BL-01, BL-09(권장 — `processApproval()` 분해와 함께 진행하면 메일
  발송 호출부가 이미 각 메서드로 나뉘어 있어 템플릿 교체가 쉬움)
- **작업 내용**: 상신/승인/반려/긴급공지 4곳의 "인사말+본문+제목" 조립 패턴을 `common` 패키지의 메일
  템플릿 빌더(또는 액션별 Template Method)로 추출.
- **완료 조건**: 4개 메일 발송 케이스의 제목·본문 문자열이 기존과 동일. BL-01 테스트 green 유지.

### BL-11 — Rich Domain Model 전환
- **관련 스멜**: S12 (Anemic Domain Model), S13 (캡슐화 부재)
- **우선순위**: P3 · **선행 작업**: BL-01, BL-02, BL-09 (상태 전이 로직이 서비스에서 정리된 뒤 도메인으로
  옮기는 편이 안전)
- **작업 내용**: `Approval`/`Notice`/`Schedule` 의 불필요한 public setter를 제거하고, 상태 전이는
  `approval.submit()`/`approval.approve(userId)` 같은 도메인 메서드로만 허용. 서비스는 이 메서드를
  호출하는 얇은 오케스트레이션 계층으로 축소.
- **완료 조건**: 외부에서 setter로 직접 상태를 바꿀 수 없음을 확인하는 테스트 추가. BL-01 테스트 green 유지.
- **비고**: 범위가 넓어 다른 P1/P2 티켓이 끝난 뒤 별도 스프린트로 진행 권장.

### BL-12 — 마무리 정리(라벨 중복·네이밍·주석)
- **관련 스멜**: S14(잔여분, BL-07에서 대부분 해소), S16 (Poor Naming), S17 (Comment Smell)
- **우선순위**: P3 · **선행 작업**: 특별히 없음(위 티켓들 진행 중 자연스럽게 함께 처리 가능)
- **작업 내용**: `d`, `u`, `s`, `proc`, `tmp`, `flag1` 등 약어 변수·파라미터를 CLAUDE.md 네이밍 컨벤션에
  맞게 전부 Rename. 이름 개선으로 무의미해진 인라인 주석 제거.
- **완료 조건**: 코드베이스에 한 글자·약어 식별자 없음(코드 리뷰로 확인). BL-01 테스트 green 유지.
- **비고**: 별도 스프린트로 묶기보다, 위 티켓들을 처리하며 지나가는 코드에 대해 즉시 적용하는 것을 권장.

## 백로그 밖 — S18 (조용한 실패)

CLAUDE.md 불변 규칙상 이번 리팩토링 백로그의 착수 대상이 아니다. BL-01에서 "권한 없음/상태 불일치 시
조용히 무시"라는 현재 동작을 테스트로 고정만 해두고, 실제 수정 여부(예외로 전환할지)는 **별도 재설계
단계**의 백로그로 넘긴다.
